# MoneyGame Fact Check

## Overview

이 스킬은 책 원고(`chapters/*.md`)의 "사실 주장"을 찾아 신빙성을 올릴 근거를 조사하고, 출처를 원고에 남긴다.

산출물은 두 가지다.
- 원고 반영: `chapters/` 내 해당 문서에 각주/주석(footnote) 추가
- 작업 로그: `status/fact_check/YYYY-MM-DD.md`

핵심 제약:
- 단계별로 진행한다. 이해/동의 없는 내용은 원고에 확정 반영하지 않는다. (`AGENTS.md`)
- 책 밖 정보는 "책 내용을 더 정확히 이해시키기 위한 보조"로만 사용한다.
- 출처는 가능한 한 1차/공신력 있는 자료를 우선한다. (정부/공공기관/학술/공식 통계/대표 지수 제공자)

## Required Inputs (read first)

- `AGENTS.md`
- `guides/WRITING_RULES.md`
- `guides/DISPLAY_GUIDE.md`
- `assets/target_audience.md`
- `status/fact_check/template.md`

## What Counts As A Fact Claim

아래 항목은 기본적으로 "검증 대상"이다.
- 연도/기간이 들어간 주장(예: 1983~2023)
- 숫자/배수/퍼센트/수익률/물가 상승 등 정량 주장
- 역사적 사실, 제도/규정, 정의가 중요한 용어 설명
- "통계적으로 증명" 같은 표현이 붙는 주장

## Output Conventions (Chapters)

각주 방식(GitHub Markdown footnotes)을 기본으로 한다.

예:

```md
이것은 사실 주장이다.[^src-1]

[^src-1]: 통계/보고서 제목, 발행기관, 발행연도. https://example.com
```

규칙:
- 각주는 문장 끝에 달고, 문서 하단에 `## Sources` 또는 각주 블록으로 정리한다.
- 외부 자료를 썼다면 "무엇을 보조하기 위한 링크인지" 1줄로 적는다.
- 상대 경로로 `chapters/` 밖 파일을 링크할 때는 문서 위치에 맞는 상대 경로를 쓴다.

### Footnote ID Policy (dedupe)

각주 ID는 중복을 최소화하기 위해 아래 규칙으로 만든다.
- 기본 형식: `[^fc-<chapter>-<n>]`
- `<chapter>`: 파일명 선두 번호 2자리(예: `01`, `10`)
- `<n>`: 해당 챕터 내 순번(001부터)

예:
```md
...이다.[^fc-01-001]

[^fc-01-001]: ...
```

중복 방지:
- 이미 동일한 출처를 같은 챕터에서 썼다면 새 각주를 만들지 말고 기존 각주 ID를 재사용한다.
- 동일 출처라도 "보조 목적"이 다르면(무엇을 뒷받침하는지) 각주를 분리할 수 있다.

### Sources Block Policy (recommended)

가독성을 위해 챕터 하단에 `## Sources`를 추가하고, 그 아래에 각주를 모으는 방식을 권장한다.
단, 기존 챕터가 각주 없이 운영 중이라면, 처음 추가할 때는 "각주만"으로 시작하고 점진적으로 정리한다.

## Source Quality Policy (priority)

출처는 아래 순서로 우선한다. 가능하면 1~3을 목표로 한다.

1. 1차/공식 통계: 정부/공공기관/중앙은행/국가통계/국제기구(예: OECD, IMF, World Bank)
2. 대표 지표 제공자/공식 문서: 거래소/지수사업자/공식 기관 보고서
3. 학술/전문서: 피어리뷰 논문, 대학/연구기관 보고서, 공신력 높은 출판사
4. 주류 언론/리서치: 1~3을 인용하고 원문 링크가 명확한 경우에 한해 보조로 사용
5. 기타(블로그/개인 글): 최후 수단. 원고에 넣기 전 사용 이유를 로그에 남긴다.

금지(원칙):
- 출처가 불명확한 캡처/짤/커뮤니티 글을 "사실 근거"로 사용하지 않는다.

## Disagreement Policy (when sources conflict)

출처 간 값/정의가 불일치할 때는 "정답 만들기"보다 "검증 가능한 문장"을 우선한다.

처리 규칙:
- 숫자 단정 -> 범위/조건부 표현으로 완화(예: "약", "대략", "당시 기준", "자료에 따라 차이가 있다")
- 비교/배수 단정 -> 기준 연도/품목/측정 방식 명시
- 불일치가 해소되지 않으면: 문장 자체를 삭제/완화하는 안을 먼저 제안한다.

리포트/로그 규칙:
- 어떤 출처가 무엇에서 불일치하는지(`정의`, `기간`, `표본`, `환산 기준`)를 `status/fact_check`에 기록한다.

## Research Mode (online vs offline)

이 스킬은 원칙적으로 외부 리서치가 필요하다.

- 온라인 리서치 가능:
  - 주장별 출처를 찾아 각주를 제안한다.
- 온라인 리서치 불가(또는 제한):
  - 검증 대상 주장 목록과 "출처 필요" 마커만 만들고, 각주는 `TBD`로 남긴다.
  - 이 경우 chapters에 확정 각주를 추가하지 않고 `status/fact_check`에 대기 항목으로 기록한다.

## Output Conventions (Status Log)

`status/fact_check/YYYY-MM-DD.md`에 다음을 기록한다.
- 범위(어떤 챕터/섹션)
- 검증한 주장 목록(요약)
- 채택한 출처 목록
- 원고 반영 여부(반영/대기/보류)와 이유
- 다음 액션(가장 작은 1단계)

## Workflow (Step-by-step)

1. 주장 추출(자동 + 수동)
   - 수치/연도/비교/배수 주장 후보를 뽑는다.
   - "독자가 사실로 받아들이는 문장"을 우선한다.

2. 리서치 계획 작성(원고 변경 전)
   - 검증할 주장 리스트와 후보 출처 타입을 적는다.
   - 신뢰도 낮은 출처(블로그/카페/무명 사이트)는 최후 수단으로 둔다.

3. 리서치 수행
   - 각 주장에 대해 최소 1개 이상의 근거를 찾는다.
   - 출처 간 불일치가 있으면 "불일치"로 기록하고, 원고 문장을 보수적으로 다듬는 방향을 제안한다.

4. 반영 제안(기본)
   - 바로 원고를 고치지 말고, 어떤 문장에 어떤 각주를 달지 제안한다.
   - 사용자가 동의하면 그때 chapters를 수정한다.

5. chapters 반영(승인 후)
   - 각주/주석 추가
   - 필요하면 문장도 "검증 가능한 형태"로 보수적으로 수정

6. status 로그 저장
   - 그날 작업을 `status/fact_check/YYYY-MM-DD.md`로 남긴다.
